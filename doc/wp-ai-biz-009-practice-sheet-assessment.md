# WP-AI-BIZ-009: 练习卷生成运行时化评估与接入

## 评估结论

**决策：暂不纳入统一运行时，保持规则链独立，但预留 AI 增强接口**

---

## 1. 当前状态分析

### 1.1 练习卷生成架构

```
PracticeSheetService::generate_practice_sheet()
    │
    ├─→ 读取错题记录（规则筛选）
    ├─→ 选择题目模板（规则匹配）
    ├─→ 应用变式规则（规则变换）
    └─→ 组装练习卷（纯规则，无 LLM 参与）
```

**关键文件**：
- `services/practice_sheet.rs` - 规则链实现
- `pages/PracticeSheetsPage.tsx` - 前端页面

### 1.2 与运行时的关联分析

| 维度 | 练习卷生成 | 运行时能力 |
|------|-----------|-----------|
| **核心逻辑** | 规则筛选 + 模板匹配 | LLM 生成 + Agent 编排 |
| **输入** | 错题记录 ID 列表 | 自然语言 Prompt |
| **输出** | 结构化练习卷（确定性的） | 自然语言/JSON（概率性的） |
| **确定性** | 高（规则可预测） | 中（LLM 需人工确认） |
| **延迟要求** | 秒级（规则计算快） | 秒级~分钟级（LLM 推理） |
| **人审闭环** | 当前无需确认 | 运行时强制要求确认 |

### 1.3 业务场景对比

**练习卷生成场景**：
- 教师选择学生 + 知识点范围
- 系统根据错题率、难度分布规则筛选题目
- 输出固定格式练习卷（PDF/Word）
- 教师可直接打印发放

**运行时适用场景**：
- 需要 AI 生成个性化题目描述
- 需要 AI 根据学生特点调整措辞
- 需要多轮对话确定题目难度

---

## 2. 纳入/不纳入的权衡

### 2.1 纳入运行时的优势

1. **统一体验**：所有 AI 功能走同一套交互流程
2. **可扩展性**：未来可加入 AI 题目改写/生成
3. **审计统一**：执行记录统一进入 ExecutionStore

### 2.2 纳入运行时的劣势

1. **过度设计**：纯规则业务引入 LLM 编排增加复杂度
2. **延迟增加**：运行时初始化 + LLM 调用增加响应时间
3. **人审负担**：运行时强制确认，打断教师快速生成流程
4. **改造成本高**：需重构整个练习卷生成链

### 2.3 关键决策因素

> **练习卷生成的核心是"规则组合"而非"AI 生成"**

当前规则链已满足业务需求：
- 错题筛选规则明确
- 题目模板固定
- 变式逻辑确定

**AI 增强点（未来可接入）**：
- 题目描述个性化改写
- 根据学生心理特点调整措辞
- 生成家长指导建议

## 3. 推荐方案：混合架构

### 3.1 核心决策

**练习卷主链保持规则链，但增加 AI 增强接口**

```
PracticeSheet Generation
    │
    ├─→ Rule Engine（规则链 - 主流程）
    │       ├─→ 错题筛选
    │       ├─→ 模板匹配
    │       └─→ 变式应用
    │
    └─→ AI Enhancement（运行时 - 可选增强）
            ├─→ 题目描述改写
            ├─→ 个性化措辞
            └─→ 家长指导生成
```

### 3.3 架构边界明确化

**练习卷服务边界**：
- `PracticeSheetService` - 纯规则链，不依赖运行时

**数据模型边界**：
- `PracticeSheet` 表 - 规则生成结果

## 5. 结论

| 项目 | 决策 |
|------|------|
| **练习卷主链** | ❌ 暂不纳入运行时，保持规则链 |
| **AI 增强** | ✅ 预留接口，按需接入运行时 |
| **文档化** | ✅ 代码注释 + 架构文档明确边界 |
| **重新评估时机** | 当 AI 题目生成需求明确时 |

**理由**：
1. 练习卷生成的核心能力是规则组合，非 AI 生成
2. 当前规则链已满足业务需求，改造成本高
3. 运行时的人审闭环会打断教师快速生成流程
4. 保留 AI 增强扩展点，未来可按需接入
